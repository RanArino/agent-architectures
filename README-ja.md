# エージェント・アーキテクチャ

*Read this in English: [README.md](README.md)*

この PoC が問うのは──**同じタスクを「単一エージェントのループ」「ワーカーを隔離するオーケストレーター」「ハンドオフ」の3通りに通すと、何が変わるのか?** という一点です。仮説は、3つのフローを並べて走らせれば、図で説明するよりも「**制御がどこにあるか (control location)**」と「**コンテキストを誰が所有するか (context ownership)**」が直感的に理解できる、というものです。

実際の OpenAI のモデル呼び出しと、本物の OpenAI Agents SDK のプリミティブを使います。ただし2つの検索ツールはローカルの固定テキストを返すだけです。外部の検索サービスを呼ぶことは一切ありません。

## 前提条件

- Python 3.11 以上
- [`uv`](https://docs.astral.sh/uv/)
- アーキテクチャ実行用の `OPENAI_API_KEY`（`tasks` コマンドとテストスイートはキー不要）
- 任意の `OPENAI_MODEL`。デフォルトは `gpt-4.1-mini`

### API キーを設定する

サンプルファイルをコピーして自分のキーを記入します（`.env` は gitignore 済み）:

```bash
cp .env.example .env
# その後 .env を編集して OPENAI_API_KEY を設定
```

`agentflow` と `agentflow-ui` はどちらも `.env` を自動で読み込みます。お好みで
`export OPENAI_API_KEY="..."` をシェルで設定してもよく、その場合はシェルの値が `.env` より優先されます。

## 実行する

このディレクトリから:

```bash
uv sync --extra dev
uv run agentflow tasks
uv run agentflow single-loop plain
uv run agentflow orchestrator single-source
uv run agentflow handoff compare
uv run agentflow compare
```

タスク引数には `plain` / `single-source` / `compare`、または自由記述の質問を渡せます。
`compare` は **9回のライブ・モデル実行**を行います（各アーキテクチャ × 各プリセット）。各実行では、
SDK のモデル呼び出しごとに送られた「フィルタ後の入力」、その応答単体のトークン使用量、推定コスト、
モデルのレイテンシを表示します。同一エージェントの複数回のモデル呼び出しはエージェント単位で集計され、
ネストしたワーカーの分はマネージャーの合計には含めません。デフォルトモデルのコスト推定は、2026-06-24
時点で確認した[公開された GPT-4.1 mini のトークン単価](https://platform.openai.com/docs/models/gpt-4.1-mini)
を用い、キャッシュトークン割引は安全側に倒して無視しています。`OPENAI_MODEL` が未知のモデルの場合コストは
`n/a` と表示されます。最終的に正なのはプロバイダーの請求額です。

キー不要のチェックはすべて次で実行できます:

```bash
uv run pytest -q
```

## 独自タスクを試す

プリセットのタスク（`plain` / `single-source` / `compare`）は意図的に短い一問一答形式にしています。
制御構造の違いを見るには十分ですが、オーケストレーターの隔離メリットが本当に効いてくる長さではありません。
並列委譲やサマリー圧縮が意味を持つのは、シングルループのコンテキストが膨れ始めてから、です。

**手っ取り早い方法 — 自由記述の質問をそのまま渡す:**

```bash
uv run agentflow single-loop "シンボリック AI とニューラルネットワークの違いを複数の観点から調べて。"
uv run agentflow orchestrator "シンボリック AI とニューラルネットワークの違いを複数の観点から調べて。"
uv run agentflow handoff "シンボリック AI とニューラルネットワークの違いを複数の観点から調べて。"
```

**より踏み込む方法 — `src/agent_architectures/fixtures.py` にプリセットを追加する:**

```python
Task("my-task", "ここに長めのマルチステップな質問を書く。"),
```

追加後、`uv run agentflow compare my-task` で3つのアーキテクチャを同じ入力で走らせて比べられます。

**どんなタスクが違いを際立たせるか:**

| タスクの形状 | 何を浮き彫りにするか |
|---|---|
| 短い・単一ソース | 3つはほぼ同じに見える。シングルループがオーバーヘッド最小 |
| 複数ソース・比較型 | ワーカー隔離が効き始める。シングルループは生のツール結果が積み上がる |
| 長い・多段階合成 | シングルループのウィンドウが最も速く膨れる。オーケストレーターのサマリーがマネージャーウィンドウを小さく保つ。ハンドオフの input_filter が有意味になる |

現在の PoC の正直な限界：ツール結果はローカルの固定テキストなので、トークン量はどれも小さいままです。
本当の比較をするには、固定ツールをライブ検索に差し替え、複数往復が必要な質問を与えてください。

## ビジュアル UI（ローカル）

ローカルの Web UI が各アーキテクチャを**スイムレーン形式のタイムライン**として描画します──コンテキスト
ウィンドウがどこで膨らむか、エージェント間で何が行き来するか（ワーカーの要約／ハンドオフ）、そして
最終回答を誰が所有するか。マルチエージェント・オーケストレーションに初めて触れる人向けです。
バックエンド（`web.py`）は CLI とは別の第2のコンポジションルートで、本物のユースケースを
データ収集用トレースとともに実行します。フロントエンドは単一の静的 `index.html`
（バニラ JS、ビルド工程なし）で、バックエンドから配信されます。

```bash
uv sync --extra ui
uv run agentflow-ui           # http://127.0.0.1:8000
```

ライブ実行には `OPENAI_API_KEY` が必要です（上記のとおり `.env` に設定）。キーが無くてもページ自体は開けます。

タスクを選び、いずれか1つのアーキテクチャを実行するか、**3つすべてを実行（compare）** を押します。
`127.0.0.1` のみにバインドします。`AGENTFLOW_UI_HOST` / `AGENTFLOW_UI_PORT` でホスト／ポートを上書きできます。

## 実装内容

| コマンド | 制御 (Control) | コンテキストの所有 | プリミティブ |
|---|---|---|---|
| `single-loop` | 1つのエージェント | 1つのウィンドウに生のツール結果が蓄積する | 生の OpenAI クライアント |
| `orchestrator` | マネージャーが制御を保持 | ワーカーは構造化タスクを受け取り、要約を返す | `Agent.as_tool()` |
| `handoff` | トリアージが制御を移譲 | 受け手がフィルタ済みの会話を継承し、最終回答を出す | `handoff()` + `input_filter` |

`single-loop` は「モデル → ツール → モデル」のターンを明示的に回します。`orchestrator` はワーカーツール
だけを所有し、各ソースワーカーは自分のソースツール1つだけを所有します。委譲には構造化された
`TaskDescription`（`objective`、`output_format`、`allowed_tools`、`boundaries`）を使います。`handoff` フローは
ソース別スペシャリストに加えて2ソース対応の比較スペシャリストを持ち、制御が境界を越える際にツールの
やり取りを取り除きます。

アプリケーションのコアは、小さな LLM / ツール / トレース / メトリクス / オーケストレーションの各ポートに
依存します。生の OpenAI クライアントと Agents SDK はアダプター層に閉じ込められ、`cli.py` が
コンポジションルートです。この境界によって、素のループのモデルプロバイダーは差し替え可能になります。
正直な限界として、M2/M3 のフローの大半は SDK アダプターが持っています──SDK 自身がオーケストレーター
だからです。

公式リファレンス: [Python Agents SDK](https://openai.github.io/openai-agents-python/)、
[ツールと agents-as-tools](https://openai.github.io/openai-agents-python/tools/)、
[ハンドオフと input filter](https://openai.github.io/openai-agents-python/handoffs/)、
[usage](https://openai.github.io/openai-agents-python/usage/)。

## 結論

**「結論の出た学習用アーティファクト」として保持する。このまま本番に昇格させない。** これは最小限の
決定的なツールで、意図したアーキテクチャ間の対比を示すものです。実機能を作る際は、ここで学んだ
制御／コンテキストの選択を再利用すべきであり、この PoC の固定データ・プロンプト・CLI をコピーすべき
ではありません。

## このPoCで分かったエージェント・フローのこと

エージェント・ループは小さい──モデルが行動を選び、アプリがそれを実行し、結果が次のモデル入力になる。
マルチエージェント・アーキテクチャが主に変えるのは、「**誰が次の行動を選ぶか**」と「**誰がどの履歴を
見られるか**」だけです。agents-as-tools は、マネージャーが制御を保持し続け、要約が狭いインターフェース
を形づくるため、**裏方の作業**に強く向きます。ハンドオフは、受け手が最終回答に責任を持つため、
**ユーザー対面で所有権を引き継ぐ場面**に向きます。コンテキストの隔離は有用ですが、ただではありません
──委譲プロンプトや要約は、設計し観測すべき「契約」になります。
